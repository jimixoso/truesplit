PRD: "TrueSplit" — An Immutable, Idempotent Group Ledger1. One-linerTrueSplit is an ACID-compliant, double-entry expense settlement engine for shared living and travel groups that guarantees a double-clicked "Submit" button on a flaky 3G network will never double-charge a user.2. Problem & Target UserThe Specific Pain: A 10-person bachelor party splitting 45 rapid-fire shared expenses over a weekend. When standing in a crowded bar with 1 bar of cellular service, Bob taps "Submit $340 for dinner", sees a hanging loading spinner, panics, and taps it two more times. Standard apps log this as $1,020 owed. Fixing "ghost duplicates" requires manual group-chat forensics, destroys ledger trust, and creates friction among friends.Why existing solutions fail: Mainstream consumer settlement apps (Splitwise, Tricount) treat expenses as simple, mutable database rows (UPDATE balances SET amount = amount + 340). They prioritize client-side UI optimism over strict API-level idempotency, meaning an ungraceful TCP drop during a POST handshake routinely results in duplicate database inserts.3. ScopeIn scope (MVP):Double-entry ledger core (every single expense explicitly forces balanced Debit and Credit legs).Idempotent transaction ingestion engine guarded by an X-Idempotency-Key header.Server-Sent Events (SSE) live-streaming ledger updates to the client."The Chaos Switch": A UI component that intentionally fires 10 perfectly identical, concurrent POST requests to test the idempotency lock live.Greedy debt-simplification algorithm (minimizing total peer-to-peer edges).Explicitly out of scope:Real bank/ACH integration: We are proving the math and state machine; faking the final Stripe/Plaid clearing step keeps us focused on the hard distributed systems problem.Multi-currency / live FX tables: Locked exclusively to USD stored as integers (cents). Introducing floating-point FX drift ruins strict zero-sum ledger invariants.Receipt OCR scanning: High-risk hackathon time-sink that distracts judges from the core concurrency achievement.Dynamic User Auth: Hardcoded to 4 static mock profiles (usr_alice, usr_bob, usr_charlie, usr_david) via a simple "Switch User" dropdown.4. Core Entities & Data ModelAccountid (UUIDv4, PK)group_id (UUID, FK)owner_id (String — e.g., usr_alice)type (Enum: LIABILITY_MEMBER, ASSET_CLEARING_HOUSE)Transactionid (UUIDv4, PK)group_id (UUID, FK)idempotency_key (String, UNIQUE INDEX)description (String)status (Enum: PENDING, COMMITTED, FAILED)created_at (Timestamp)LedgerEntry (The Double-Entry Legs)id (UUIDv4, PK)transaction_id (UUID, FK)account_id (UUID, FK)amount_cents (BigInt, Strictly non-negative)entry_type (Enum: DEBIT, CREDIT)The Absolute Invariants:The Zero-Sum Law: For any committed Transaction, sum(Debit entries) == sum(Credit entries). If this evaluates to anything other than 0, the database transaction must throw a panic and roll back.The Highlander Key Rule: For any given (group_id, idempotency_key) pair, there can only ever exist one database row where status = COMMITTED.5. Key Technical Risks & Failure ModesRiskSpecifically why it's hardHandled or Deferred?The "Double-Tap" Race ConditionTwo identical requests arrive at the API gateway 2 milliseconds apart; both bypass the Redis EXISTS check before either sets it.Handled: By leaning on Postgres as the ultimate source of truth via a UNIQUE(group_id, idempotency_key) constraint. The 2nd request hits a duplicate key value violates unique constraint DB error, which our app catches and maps to a "Wait and fetch cached result" loop.Partial Ledger WritesThe server writes Alice’s +$30.00 credit leg, but the pod runs out of memory and dies before writing Bob’s -$30.00 debit leg.Handled: Using native SQL BEGIN ... COMMIT transaction blocks. If the connection drops mid-block, Postgres automatically discards the orphan leg.Payload Mutation AttackA user sends Key ABC with "Dinner: $10", then maliciously retries Key ABC with "Dinner: $500".Deferred (MVP): We assume the client sends identical payloads per key. (In the Deep Cut, we store a SHA-256 hash of the payload inside the lock and reject mismatched hashes).7. Tech Stack & RationaleLanguage: Go (Golang) * Why: The standard library's native net/http and goroutines make writing the multi-threaded "Chaos Load Tester" native to the repo. Its explicit if err != nil pattern forces you to acknowledge every single database transaction failure.Database: PostgreSQL (via pgx driver)Why: ACID is non-negotiable here. Using MongoDB or a loose NoSQL document store for double-entry bookkeeping is an architectural sin; we require strong relational CHECK constraints and SERIALIZABLE isolation levels.Caching/Mutex: RedisWhy: In-memory atomic SETNX operations are the industry standard for high-throughput distributed locks.Client: Next.js (App Router) + TailwindWhy: Speed of UI assembly. We will use vanilla EventSource to consume the Go backend's Server-Sent Events, completely bypassing the need to set up a heavy WebSocket gateway.10. Open Questions / AssumptionsThe "Check-Splitter's Remorse" problem: If Alice successfully posts an idempotent transaction, but 4 seconds later realizes she typed $40.00 instead of $400.00, how does she fix it? Assumption for MVP: The ledger is strictly append-only. She cannot edit it; she must issue a manual compensating $360.00 "Correction" transaction. 2. Clock Skew: If the group travels across time zones (e.g., flying from New York to Tokyo mid-trip), how do we sort the "Daily Ledger Feed"? Assumption: The backend ignores client local timestamps entirely and stamps every LedgerEntry strictly at the DB-server's UTC NOW() tick.
